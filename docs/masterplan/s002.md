# Сессия 002 — Core Pillars

## Контекст
Входные артефакты: все артефакты Сессии 001 (ADR-001, invariants, Directory.Build.props, logging system).
Сессия 002 фиксирует **дизайн-столпы** (core pillars) — фундаментальные правила геймдизайна и технической реализации, которые не пересматриваются в последующих фазах.

## Ограничения
- Все правила из MasterPlan (1–8) по-прежнему действуют
- Столпы должны быть проверяемыми (каждый pillар → тест или инвариант)
- Изменение pillar требует отдельного ADR и согласования с Мастером

## Core Pillars (Дизайн-столпы)

### Pillar 1: Archive-Centric World
> Мир организован как Архив с иерархической адресацией. Всё исследование, контент и прогрессия привязаны к Архиву.

- Адрес: Sector → Hall → Module → Shelf → Tome → Page
- Каждый уровень генерируется детерминированно из seed
- Архив — не декорация, а основная механика навигации и прогресса

### Pillar 2: Deterministic Generation
> Одинаковый seed всегда генерирует одинаковый мир. Нет скрытой рандомизации.

- Seed передаётся через ArchiveAddress
- Генератор не использует DateTime, Thread-local, внешние API
- Replay-тестирование обязательно

### Pillar 3: State Sovereignty (WorldState SSOT)
> WorldState — единственное место хранения мутабельного состояния мира. Никакие другие объекты не дублируют состояние.

- UI только читает WorldState
- Мутация только через Runtime-команды
- Save/Load оперируют только WorldState

### Pillar 4: Combat-Exploration Duality
> Исследование Архива и боевая система — два равноправных режима с общим состоянием.

- Переход exploration ↔ combat сохраняет контекст в WorldState
- Бой не «отдельная сцена», а часть непрерывного мира
- Результат боя влияет на состояние Архива

### Pillar 5: Mission-Driven Progression
> Прогресс игрока определяется миссиями. Каждая миссия — граф MissionNode с условиями.

- MissionDefinition → MissionNode graph
- Валидация: достижимость, dead-end safety, cycle safety
- Миссия не запускается без прохождения валидации

### Pillar 6: Layered Architecture Contract
> Три слоя (Core → Runtime → UI) с однонаправленными зависимостями. Нарушение — блокирующий дефект.

- Core: нулевые внешние зависимости
- Runtime: зависит только от Core
- UI: зависит от Core + Runtime, но не мутирует состояние напрямую

### Pillar 7: Content as Data
> Игровой контент (миссии, баланс, шаблоны архива) описывается как данные (JSON/YAML), а не hardcode.

- Контент валидируется автоматически до запуска и до merge
- Схемы контента живут в BabylonArchiveCore.Content
- Баланс изменяется без перекомпиляции
