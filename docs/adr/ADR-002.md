# ADR-002: Фиксация Core Pillars (дизайн-столпы)

## Статус
Принято — Сессия 002

## Контекст
После установки архитектурного фундамента (ADR-001) необходимо зафиксировать **неизменяемые дизайн-столпы** — правила, определяющие идентичность игры Babylon Archive Core. Эти pillar'ы не пересматриваются без отдельного ADR и согласования с Мастером.

## Решение
Принято 7 Core Pillars:

### P1: Archive-Centric World
Мир = Архив. Иерархическая адресация Sector→Hall→Module→Shelf→Tome→Page определяет пространственную и контентную навигацию. Архив — не декорация, а основная ось прогрессии.

### P2: Deterministic Generation
Seed-детерминизм: `generate(seed) == generate(seed)` всегда. Нет скрытого рандома (DateTime, Thread-local). Обязательное replay-тестирование.

### P3: State Sovereignty (WorldState SSOT)
WorldState хранит всё мутабельное состояние. UI читает, Runtime мутирует через команды. Save/Load работают только с WorldState.

### P4: Combat-Exploration Duality
Исследование и бой — равноправные режимы с единым состоянием. Переход exploration↔combat непрерывен. Результат боя влияет на Архив.

### P5: Mission-Driven Progression
Прогрессия через миссии: MissionDefinition → граф MissionNode. Обязательная валидация (достижимость, dead-end, циклы) до запуска.

### P6: Layered Architecture Contract
Core→Runtime→UI с однонаправленными зависимостями. Нарушение = блокирующий дефект. Принудительно через Directory.Build.props.

### P7: Content as Data
Контент описывается как данные (JSON/YAML), не hardcode. Валидация до запуска и до merge. Баланс без перекомпиляции.

## Альтернативы
| Вариант | Причина отклонения |
|---------|-------------------|
| Мягкие рекомендации вместо pillar'ов | Размывание правил на длинной дистанции (150 сессий) |
| Больше pillar'ов (10+) | Избыточная жёсткость; 7 покрывают ключевые области |
| Pillar'ы без тестовых проверок | Непроверяемые правила — мёртвый документ |

## Риски
- **R4**: Pillar P4 (Combat-Exploration) может усложнить ранние прототипы — митигация: начать с exploration only, добавить combat в Блоке 03
- **R5**: Pillar P7 (Content as Data) требует инфраструктуры валидации — митигация: ContentValidators в tools/
- **R6**: Жёсткость pillar'ов может блокировать адаптацию — митигация: изменение через отдельный ADR

## Следствия
- Каждый pillar получает инвариант в `docs/architecture/invariants.md`
- Тесты Session002 фиксируют проверяемость pillar'ов
- Все последующие сессии сверяются с pillar'ами перед принятием решений
- Pillar'ы включаются в review checklist каждой PR
