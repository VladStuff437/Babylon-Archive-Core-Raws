# ADR-001: Трёхслойная архитектура с однонаправленными зависимостями

## Статус
Принято — Сессия 001

## Контекст
Babylon Archive Core — RPG с roguelike-элементами, боевой системой, дипломатией и исследованием процедурно генерируемого Архива. Проект требует чёткого разделения ответственности для масштабирования на 150 сессий разработки.

## Решение
Принята трёхслойная архитектура с однонаправленными зависимостями:

```
Core  ←  Runtime  ←  UI
```

### Слои
1. **Core** (`BabylonArchiveCore.Core`) — доменные модели, контракты, чистое состояние.
   - WorldState (SSOT), PlayerState, MissionDefinition, MissionNode, ArchiveAddress
   - Не зависит ни от чего, кроме BCL
2. **Runtime** (`BabylonArchiveCore.Runtime`) — игровой цикл, генерация, боевая система, ИИ.
   - Зависит только от Core
3. **UI** (`BabylonArchiveCore.UI`) — представление, ввод, HUD, экраны.
   - Зависит от Core и Runtime
   - **Запрещено**: обратные зависимости UI → Core для мутации состояния

### Инварианты
- WorldState является единственным источником правды для мутабельного состояния
- UI не мутирует WorldState напрямую — только через Runtime-команды
- Seed-детерминизм: одинаковый seed → одинаковый результат генерации

## Альтернативы
| Вариант | Причина отклонения |
|---------|-------------------|
| Двуслойная (Logic + UI) | Недостаточная изоляция доменных моделей от игровой механики |
| ECS-first | Преждевременная оптимизация; может быть введена в Runtime позднее |
| Microservices | Излишняя сложность для single-player RPG |

## Риски
- **R1**: Слой Runtime может «распухнуть» — митигация: выделение sub-modules (Combat, Generation, AI)
- **R2**: Сериализация WorldState при save/load — митигация: Session 008 выделена под Save/Load v1
- **R3**: UI-логика может просочиться в Runtime — митигация: code review + Directory.Build.props constraints

## Следствия
- Каждый слой живёт в отдельном проекте .csproj
- Directory.Build.props запрещает обратные ссылки
- Тесты для Core не зависят от Runtime/UI
