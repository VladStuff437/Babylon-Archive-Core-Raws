# ADR-006: ADR как формальный контракт

**Статус**: Принято
**Дата**: Сессия 006
**Контекст**: ADR-001..ADR-005, decision-log

## Решение

Формализовать ADR-процесс как обязательный контракт разработки:

### ADR Lifecycle
```
Proposed → Accepted → [Active]
                    ↘ Superseded (by ADR-XXX)
                    ↘ Deprecated (reason)
```

### Обязательные секции ADR
1. **Статус** — Proposed / Accepted / Superseded / Deprecated
2. **Дата** — сессия создания
3. **Контекст** — зависимости на другие ADR и INV
4. **Решение** — что принято, с деталями
5. **Альтернативы** — что рассматривалось и почему отклонено
6. **Риски** — что может пойти не так + митигация
7. **Связи** — Реализует (INV), Зависит от (ADR), Влияет на (сессии)

### Traceability Chain
```
ADR → INV → Code (contracts/impl) → Tests
```
Каждый ADR должен быть привязан к минимум одному INV. Каждый INV — к тестам.

### Нумерация
- ADR-NNN: порядковый номер, совпадает с номером сессии создания
- При supersede: старый ADR получает ссылку на новый

## Альтернативы
1. **Неформальные заметки**: Отклонено — теряются, нет traceability
2. **RFC-процесс с ревью**: Отклонено — overhead для solo-разработки
3. **ADR без lifecycle**: Отклонено — устаревшие решения создают путаницу

## Риски
- ADR-bloat: слишком много ADR → митигация: INDEX.md с фильтрацией по статусу
- Orphaned ADR без INV → митигация: INV-011 требует связку

## Связи
- Реализует: INV-011
- Зависит от: все предыдущие ADR
- Влияет на: все последующие сессии (ADR workflow)
